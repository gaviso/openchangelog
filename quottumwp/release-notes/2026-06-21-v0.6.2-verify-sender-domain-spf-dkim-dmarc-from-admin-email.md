The DKIM chip on \`/admin/email\` is no longer just decoration — there's a **Verify** button next to it that does a live DNS check and tells you what's missing.

### What it does

POST \`/api/admin/email\` with \`{action: 'verify'}\` resolves the saved sender domain via three TXT lookups:

- **SPF** — root domain, must contain \`v=spf1\`
- **DKIM** — probes common selectors. Resend uses \`resend._domainkey\` / \`default._domainkey\`. Generic SMTP probes \`default\` / \`google\` / \`k1\` / \`mxvault\` / \`s1\`. Passes on the first selector that has a \`v=DKIM1\` record.
- **DMARC** — \`_dmarc\` subdomain, must contain \`v=DMARC1\`

The DKIM verdict is persisted so the chip stays accurate across reloads. Every verify call writes an audit row with per-record results.

### UI

The form gets:

- **Verify button** next to the DKIM chip (only when a sender domain is saved)
- A **results panel** with ✓/✗ per record class and the resolved TXT contents
- When DKIM fails, a one-line hint tells the operator **exactly what record to add and at which selector** — different copy for Resend (add the record Resend showed you in domain setup) vs SMTP (your provider issues a DKIM record; add it as TXT at their specified selector)

### Also

The chip is no longer Resend-only — DNS auth checks apply to any provider, including raw SMTP servers and self-hosted mailservers.
