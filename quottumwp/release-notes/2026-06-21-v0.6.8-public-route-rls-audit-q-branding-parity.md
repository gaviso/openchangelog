Follow-up audit after v0.6.7's invite fix surfaced two more silent-data bugs in the public token-authorized routes. Both look like the v0.5.0 hotfix pattern: embedded JOINs to members-only-RLS tables silently return \`null\`, so data quietly fails to render with no error.

### Bugs fixed

**\`/q/[token]\` — public quote**
- \`clients(...)\` embed stripped to \`null\` → \"Prepared for {company}\" line never rendered
- Also missing **all workspace branding**: hardcoded \"Quottum\" wordmark, no uploaded logo, no terms block, no workspace name, no footer text. \`/c/[token]\` got these in v0.5.0; \`/q/\` was never updated.

**\`/c/[token]\` — public contract**
- Same \`clients(...)\` RLS strip → \"Issued to {company}\" line + footer's recipient email silently missing

### What changed

Both pages now use the same pattern as the v0.5.0 hotfix: separate service-role lookups for workspace_settings, workspaces, and clients. Token is the access control, only safe-to-public fields are selected.

\`/q/[token]\` specifically gained parity with \`/c/[token]\`:
- Header: workspace logo (or Quottum mark fallback) + workspace name
- Body: client name + contact line at the top
- Terms & Conditions block rendered after actions
- Branded footer reading from \`workspace_settings.footer_text\`
- Currency cascade: quote → workspace default → USD
