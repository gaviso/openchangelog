The complete feature-enforcement audit. The v0.7.197 sweep found 26 catalog features that operators could grant/block via the feature-overrides editor but which **no code actually consulted** — a block landed in the DB and did nothing. This release closes the entire backlog.

### Method: plan-inclusion data over audit severity

The audit's severity tiers conflated "core feature" with "paid feature." Before writing any gate, each candidate was checked against `plan_features` — does the **Free** plan include it? That single question decided everything and corrected the audit repeatedly:

- **Saved the free tier** from gating `quote_builder`, `quote_pdf`, `public_quote_view`, `dashboard_analytics`, `client_management`, `client_detail` — all universal (Free includes them by design). Gating those would have broken core flows.
- Confirmed the genuinely paid-only features that needed gates.

Net: ~11 real gates, not 26.

### Gates added

All are **downgrade-path** protections — `contract_management` / `team_members` already block creation on Free, so these only bite when a paid workspace downgrades and keeps using what it no longer pays for.

| Feature | Surfaces gated |
|---|---|
| `contract_lifecycle` | PATCH /api/contracts/[id] (terminate/archive/restore/edit); GET + DELETE stay open |
| `contract_pdf` | /contracts/[id]/pdf page + Download-PDF button |
| `public_contract_view` | /c/[token] public contract page → blocked view |
| `revenue_metrics` | Dashboard TCV/MRR/ARR cards → upsell; register TCV/MRR columns hidden |
| `webhooks` | PATCH /api/settings (config), settings UI, /api/cron/test-event, **/api/cron/renewals dispatch** (the recurring leak) |
| `line_item_library` | POST/PATCH line-items API |
| `discount_codes` | coupon preview + coupon create |
| `renewal_reminders` | renewals cron skips the reminder pipeline |
| `workspace_branding` | PATCH /api/settings branding fields + branding page upsell |
| `quote_activity_log` | quote detail Activity section → upgrade note |
| `roles_permissions` | member role-change PATCH + permissions page banner |

### Two product decisions baked in

- **`terms_conditions` → Free.** Was Pro/Business-only; operator chose to grant it to everyone rather than gate it (migration 089). Now universal.
- **`automated_emails` = branded styling, not send-at-all.** Emails always send (Free keeps the link-delivery flow). Pro/Business get **their own logo** in the email; Free keeps the Quottum template. New public route `/api/branding/logo/[workspaceId]` decodes the stored data-URI logo and serves it as real image bytes so it renders in Gmail/Outlook (which strip data-URI `<img>`). `sendEmail` swaps the logo for entitled workspaces. Pure code, no migration.

### Gate design conventions

- **Server gate is the authority**; UI affordances (disabled buttons, upsell banners, blocked views) are secondary.
- **Clearing/removing is always allowed** even when setting is gated — a downgraded workspace can turn off webhooks, strip branding, etc.
- **Read/cleanup stays open**; only paid *mutation*/management is gated.

### Migrations

088 (block-email expiry) + 089 (terms→Free) already applied to prod via MCP. No apply step at release.

### Post-release verification (recommended)

These gates haven't been exercised end-to-end against a real Free workspace (downgrade scenario). Suggested follow-up: flip a test workspace's entitlements and confirm each gate bites.

### Process note

Built largely via parallel background agents — discovery, multiple gate batches, and commits ran concurrently with pre-flight conflict checks and explicit-path staging to keep the work isolated. A final unified type-check confirmed all the parallel work composed cleanly.
