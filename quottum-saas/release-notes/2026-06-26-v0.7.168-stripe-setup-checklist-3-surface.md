Operators kept asking what permissions the Stripe webhook signing secret needs — but that question conflates three different Stripe primitives. This release adds an in-app setup checklist + matching docs that clarify all three and ship the same canonical content across three surfaces (in-app, repo, Notion).

## The misconception, untangled
- **Webhook signing secret** (`whsec_...`) — pure HMAC key for payload authenticity. Has **no permissions**.
- **Webhook endpoint event subscription** — the list of events Stripe pushes. **This** is what should be scoped (7 specific events for our handler).
- **API secret key** (`sk_*`) — should be a **Restricted Key** with surgical scopes, not a full-access secret.

## In-app checklist on `/admin/billing`
- Compact status pills for Test + Live: `✓` / `⚠` / `✗` per mode, based on what's configured in `admin_stripe_settings`
- Auto-expands when active-mode credentials are missing
- All 7 webhook events as copyable code chips + "Copy all"
- The 8 required Restricted Key scopes (Customers/Write, Subscriptions/Read, etc.) with reasons
- The 10 prohibited scopes spelled out (Connect, Issuing, etc.)
- Deep links to the right Stripe Dashboard URL for the **active mode**
- Customer Portal + expiring-card emails reminders

## Single source of truth + drift guard
[`src/lib/stripe-webhook-events.ts`](https://github.com/gaviso/quottum-saas/blob/v0.7.168/src/lib/stripe-webhook-events.ts) is now the canonical authority — exports `REQUIRED_WEBHOOK_EVENTS`, `REQUIRED_API_SCOPES`, `PROHIBITED_API_SCOPES`. The webhook handler has a compile-time TypeScript assertion that the switch covers every event in the canonical list — add an event without a case and the build fails.
