Collect payment on our own page via Stripe Elements instead of redirecting to hosted Stripe Checkout. **Entirely behind the `inline_checkout` feature flag (default OFF)** — zero user-visible change until an operator enables it per workspace. Flag OFF = the existing Checkout redirect, byte-identical.

### Backend
- **`POST /api/billing/subscribe`** — creates a `default_incomplete` subscription, returns `{ subscriptionId, clientSecret, mode, trialDays, publishableKey }`. `mode: 'payment'` (no trial → immediate charge, confirm the invoice PaymentIntent) vs `'setup'` (trial → no charge today, collect card via the pending SetupIntent). 404 when the flag is off.
- **`GET /api/billing/subscription-status`** — poll target; the subscription activates async via webhook, so the client polls until `active`.
- **Shared `resolveTrialDays` + `resolveOrCreateCustomer`** extracted so the new route and the existing Checkout route share one source — identical trial logic (a trial-consumed workspace gets an immediate charge, consistent with v0.7.206).
- **Webhook guard**: `customer.subscription.created` with status `incomplete`/`incomplete_expired` is now a no-op (previously it wrote `plan_status = 'past_due'`, mislabeling a never-charged Elements sub and binding the workspace to a sub that may expire). Activation fires on the `active`/`trialing` transition. Checkout path unaffected.

### Frontend
- **Inline two-step** in `UpgradeForm`: plan cards → on select, a Stripe Payment Element step (Card + Link + Apple Pay) with an order summary + "← change plan". Confirm by mode (`confirmPayment`/`confirmSetup`, `redirect:'if_required'`), then poll to activation. Only the clean Subscribe/Start-trial CTA routes inline; trial-carryover edge cases keep the redirect.
- **3DS return page** (`/dashboard/billing/processing`) for the redirect-challenge case — reads `redirect_status`, short-circuits failure, polls activation. Never claims success off the redirect alone.
- Card data enters a Stripe-hosted iframe — PCI scope stays SAQ A.

### Process
Built across 3 parallel Wave-1 agents (backend / webhook audit / integration survey) → reviewed → 1 Wave-2 agent (frontend) → reviewed → composed type-check + lint. The webhook agent caught a real bug (the `incomplete → past_due` mislabel) that would have broken billing state on every inline subscribe.

### Remaining (manual)
Test-mode browser validation (4242 card, a 3DS card, a decline card), confirming the test webhook is registered, then flipping the flag. No migration.
