Follow-up to v0.7.168. Stripe's "Configure destination" step asks for an Endpoint URL — but we'd buried that detail. Now it's a first-class element across all three setup surfaces.

## What changed

The canonical source ([`src/lib/stripe-webhook-events.ts`](https://github.com/gaviso/quottum-saas/blob/v0.7.169/src/lib/stripe-webhook-events.ts)) gains:
- `WEBHOOK_ENDPOINT_PATH = '/api/stripe/webhook'`
- `PRODUCTION_APP_ORIGIN = 'https://app.quottum.io'`
- `webhookEndpointUrl(origin?)` helper

All three surfaces derive from this single source — rename the route and the constant change ripples everywhere automatically.

| Surface | Where the URL appears |
|---|---|
| In-app `/admin/billing` checklist | New **Endpoint URL** section above the events list. Computes from `window.location.origin` so it works on staging too. Copy button included. |
| [`docs/STRIPE_SETUP.md`](https://github.com/gaviso/quottum-saas/blob/v0.7.169/docs/STRIPE_SETUP.md) | Section 2 split into 2a (Endpoint URL) and 2b (Subscribed events) — reads in the natural wizard order |
| [Notion runbook](https://app.notion.com/p/38bd0ab54594810a99b2ea3f19959b81) | New Endpoint URL section between the misconception callout and the events list |

All three include the same callout: same URL for both test and live destinations (signature verification keys off the active mode in `/admin/billing`), with a note about substituting the staging origin if deploying preview environments.
